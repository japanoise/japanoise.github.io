Dvorak

I use the Dvorak keyboard layout on an ortholinear split keyboard, specifically the keeb.io Iris rev 4. This is what the alphabetic area of my 'board looks like:

" , . P Y F G C R L
A O E U I ➰ D H T N S
: Q J K X B M W V Z

Emacsen use a goodly portion of this. Here's roughly what I expect the keys to do while holding CTRL - standard Emacs with a little tweaking:

" , . PPrevious line YYank (paste) FForward char GCancel C(Mode specific) RSearch backward LCenter line
ABeg of line OOpen line EEnd of line UKill to bol ITab ➰ DDelete char HHelp TTranspose chars NNext line SSearch forward
: QInsert raw JNewline KKill to eol X(Prefix) BBackward char MNewline WKill (cut) VPg down ZPg up

A couple of commands are off the alphas or use meta (alt) instead:

  • C-@ (control-space) - Set mark (for copying/cutting)
  • C-_ (control-forwardSlash) - Undo
  • M-w - Copy
  • M-v - Pg up
  • M-f & M-b - forward and backward word
  • M-n & M-p - next and previous paragraph
  • M-t - transpose chars
  • M-< & M-> - beg and end of buffer

Emacs keys make sense, in the same way that English pronunciation makes sense - it does, but the logic is usually unclear. Some are mnemonic (P, N, F, B, S, D, O), some are due to legacy (I, J, L, and M are all ASCII control characters) and the meta binding plays on logic, but many are just chaotic and weird. With Emacs bindings, you map your control key to a sensible position and then just learn them.

Other bindings, like WordStar and to a lesser extent Vi, are based not on mnemonics or legacy but on keyboard position. Keys that are used a lot are on the homerow or nearby, whereas keys used less are in the more awkward positions. Vi has the infamous HJKL, which it can use for movement keys as it does not use control keys (otherwise, C-j, C-l and maybe C-h would be problematic). However, when using these on a Dvorak keyboard the logic breaks down.

aoeui is an experimental editor to translate this idea to Dvorak. It uses Meta (Alt) as a modifier to avoid the problem of legacy ascii control characters. This frees up the most problematic of these, C-m, which is still what the return key outputs, and also allows the usage of more unusual bindings like M-". The layout looks like this:

"goto tag ,fold .unfold PNext window YNew window FCopy GBack Line CFore Line RBack Screen LFore Screen
A OMacro EShell UUndo I ➰ DCut HBack Char TFore Char NBack Word SFore Word
:New QSuspend or quit J KSave XVisit File BPaste M WNew view in window VSelect ZRecenter

This is really cool! Unfortunately the editor is not under active development; and switching to new bindings on Emacs is basically impossible, so it's not like I can try them there.

I thought of adopting the editor as a new maintainer, or using the bindings in a new editor (like a fork of emsys or gomacs). However, the more I looked at it, the more I wanted to tweak things with ideas from WordStar. So a new, future project was born: a terminal word processor, using an optimised Dvorak layout!

Here's the WIP for this new layout:

" , . P Y F G CScroll up RPrev line LPage up
A O E U I ➰ DBack Word HBack Word TBack Char NFore Char SFore Word
: Q J K X B M WScroll down VNext line ZPage down
